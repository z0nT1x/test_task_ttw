# Аналітична модель даних для WinWin.travel

## Опис проєкту
Цей репозиторій містить прототип аналітичної моделі для тревел-платформи WinWin.travel. Мета проєкту — спроєктувати структуру бази даних, згенерувати реалістичний набір даних та провести глибокий SQL-аналіз для пошуку точок росту бізнесу.

Модель фокусується на поведінці користувачів, ефективності персоналізації через ШІ та аналізі воронки продажів.

## Архітектура даних (Database Schema)
База даних складається з 6 взаємопов'язаних таблиць, що охоплюють повний шлях клієнта:

*   **users**: профілі користувачів, типи пристроїв та канали залучення.
*   **hotel & room**: каталог житла з характеристиками (рейтинг, pet-friendly, ціна).
*   **requests**: пошукові запити з фіксацією використання AI-асистента.
*   **activities**: поведінкові дані (лайки конкретних номерів).
*   **orders**: фінальні транзакції та деталі бронювання.

## Технологічний стек
*   **СУБД**: MySQL / MariaDB (використано DBeaver для управління).
*   **Генерація даних**: Python 3.x + бібліотека Faker (створено 1000+ унікальних записів).
*   **Аналітика**: SQL-запити з використанням функцій агрегації та обчислення часових інтервалів.

## Ключові аналітичні запити
У файлі queries.sql реалізовано 5 критичних для бізнесу звітів:

1.  **AI Conversion Impact**: порівняння конверсії запитів із використанням ШІ та без нього.
2.  **Traffic Source Efficiency**: аналіз конверсії у розрізі джерел (Instagram, Google, Facebook).
3.  **Segment ARPU**: розрахунок середнього доходу на одного унікального користувача за типом операційної системи.
4.  **Full Funnel Analysis**: метрики переходу від реєстрації до фінального замовлення.
5.  **Time-to-Booking**: визначення середнього часу прийняття рішення через TIMESTAMPDIFF.

## Основні інсайти та висновки

### Ефективність персоналізації
На поточному етапі користувачі, що не використовують ШІ, демонструють вищу конверсію.
*   **Гіпотеза**: Користувачі ШІ можуть бути більш вибагливими або використовувати складні фільтри, що подовжує цикл прийняття рішення. Необхідний додатковий UX-аудит алгоритмів ШІ.

### Сегментація за пристроями (Android vs iOS)
Найбільшу виручку та кількість трафіку генерує Android.
*   **Аналіз**: Оскільки рівень конверсії в замовлення стабільний на всіх платформах (близько 29%), технічні помилки на етапі оплати малоймовірні. Нижчий ARPU на iOS може бути пов'язаний із різницею у купівельній спроможності або специфікою видачі контенту для різних сегментів.

### Маркетингові канали
Instagram є лідером за залученням якісного трафіку.
*   **Рекомендація**: Протестувати масштабування бюджету в Instagram за умови збереження стабільної вартості залучення (CAC), розглянувши можливість перерозподілу коштів із каналів із нижчою конверсією (наприклад, Google).

### Швидкість прийняття рішення
Середній час від запиту до замовлення складає 35 секунд.
*   **Валідація**: Такий показник може свідчити про обмежену вибірку або специфіку повторних сесій (користувач повертається для покупки після попереднього ознайомлення). Рекомендується впровадити відстеження медіанного часу та провести технічну перевірку системи логування на предмет дублювання запитів.
